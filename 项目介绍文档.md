# 「我们的空间」— AI 情侣小程序 项目介绍文档

---

## 一、项目概述

**「我们的空间」** 是一款基于微信小程序的 AI 情侣陪伴应用。它不是一个普通的工具类小程序，而是一个**有"人格"的 AI 伴侣**——"小爱"，它能陪情侣聊天、帮他们记录约会、管理愿望清单、珍藏心动瞬间、立下共同约定，甚至在他们吵架时充当情感翻译官。

**核心亮点：AI 大模型 + 全栈开发 + 产品安全意识**

| 维度 | 说明 |
|------|------|
| 项目类型 | 微信小程序全栈项目 |
| 前端 | 微信小程序原生框架（WXML / WXSS / JS） |
| 后端 | Node.js Express 本地服务器 + 微信云函数（双模式） |
| AI 能力 | DeepSeek 大模型，支持 Function Calling（函数调用） |
| 实时通信 | WebSocket 流式输出 |
| 数据存储 | 内存数据库（开发）/ 微信云数据库（生产） |
| 主题风格 | 粉色可爱风（#FF7BA6） |

## 三、功能模块详细介绍

### 3.1 首页仪表盘（pages/index）

首页是用户打开小程序看到的第一个页面，设计目标是"一眼看到恋爱全貌"。

**功能点：**
- **情侣头部：** 展示双方头像、名字和在一起的天数，支持隐私模式（名字显示为"我们"）
- **纪念日倒计时：** 计算距离下一个里程碑（100天、200天、365天...）还有多少天
- **AI 每日灵感：** 每天刷新一条约会小灵感，基于历史约会和待实现愿望个性化推荐
- **小爱主动提醒：** AI 检测约会间隔超过14天、愿望积压、纪念日临近等情况，主动推送提醒
- **数据概览：** 约会次数、待实现愿望、心动瞬间、约定数量，四个数据卡片
- **成就墙：** 横向滚动的成就勋章展示，已解锁/未解锁状态
- **最近约会：** 最近3条约会记录的摘要卡片
- **环境爱心粒子：** CSS 动画实现飘浮爱心特效

**涉及技术：** 骨架屏加载、下拉刷新、CSS动画、数据统计聚合

### 3.2 AI 对话页（pages/chat）★ 核心页面

这是整个项目最复杂、最核心的页面。用户在这里与 AI 伴侣"小爱"对话。

**功能点：**
- **多对话管理：** 支持创建多个对话、切换对话、删除对话、清空对话，侧边抽屉展示对话列表
- **AI 自主推理（Agent Loop）：** 小爱不只是回答问题，它能**自主调用工具**完成任务。比如用户说"帮我记录昨天的约会"，小爱会自动调用 `add_date_record` 工具写数据库
- **多工具链：** 小爱能在一次对话中连续调用多个工具。比如"帮我做个约会总结"→ 先查约会记录 → 再查愿望清单 → 综合给出分析
- **流式输出（Streaming）：** 通过 WebSocket 实现打字机效果，AI 回复一个字一个字地显示，体验流畅
- **推理过程可视化：** 用户可以展开/收起，看到小爱"思考"的过程（思考 → 调工具 → 看到结果 → 继续思考）
- **超时回退：** WebSocket 8秒超时自动降级到 HTTP 请求，保证通信可靠性
- **快捷标签：** 预设常用指令（策划约会、写祝福、记录约会等），一键发送
- **语音输入：** 预留了录音接口
- **AI 调解（雷区3）：** 情侣吵架时发起调解，双方确认后 AI 以"情感翻译官"角色介入

**涉及技术：** WebSocket实时通信、Agent自主推理循环、流式数据处理、多对话状态管理、降级容错

### 3.3 约会记忆馆（pages/dates）

情侣的"恋爱相册"，记录每一次约会的回忆。

**功能点：**
- **双视图模式：** 时间线视图 + 照片墙视图，一键切换
- **约会统计：** 总约会次数、去过的地方数量、最常见的心情
- **照片上传：** 支持为每次约会添加多张照片
- **心情标签：** 每次约会可选情绪（甜蜜/开心/平淡/感动/吵架），不同心情对应不同颜色
- **下拉刷新、搜索过滤**

**涉及技术：** 图片上传（multipart/form-data）、照片URL处理、双视图切换、数据聚合统计

### 3.4 愿望清单（pages/wishes）

情侣共同的心愿单，一起实现。

**功能点：**
- 三个Tab切换：待实现 / 已实现 / 全部
- 愿望分类（旅行、美食、礼物、体验...）
- AI 智能贴士（小爱给每个愿望加一个实现小建议）
- 标记已实现、删除（软删除到回收站）

### 3.5 心动瞬间（pages/moments）

珍藏那些让人心动的瞬间，类似情侣专属的"朋友圈"。

**功能点：**
- 时间线展示
- 心情 emoji 标签
- 支持添加标签（tags）
- 软删除到回收站

### 3.6 约定（pages/promises）

情侣之间的承诺和约定，带打卡机制。

**功能点：**
- 两个Tab：进行中 / 已完成
- **打卡系统：** 每次打卡记录日期，计算连续打卡天数（streak）
- 约定频率设置（每天/每周/随时）
- 软删除

### 3.7 AI 恋爱报告（pages/reports）

AI 自动生成的周报/月报/年报。

**功能点：**
- AI 根据约会数据、瞬间记录、愿望进度生成温暖有爱的文字报告
- 支持保存历史报告
- 报告列表查看

### 3.8 成就殿堂（pages/achievements）

里程碑式成就系统（不是刷次数！）。

**9个成就：**
| 成就 | 条件 | 设计理念 |
|------|------|----------|
| 初遇时刻 🌸 | 记录第一次约会 | 鼓励开始记录 |
| 心心相印 ✨ | 写下第一个愿望 | 鼓励共同规划 |
| 心动定格 💝 | 珍藏第一个瞬间 | 鼓励发现美好 |
| 和好如初 🕊️ | 吵架后继续约会 | 鼓励修复关系 |
| 说走就走 ✈️ | 去过一次远方的旅行 | 鼓励共同冒险 |
| 重要的相见 🏠 | 见过家长 | 记录关系里程碑 |
| 百天纪念 💯 | 在一起100天 | 庆祝时间沉淀 |
| 周年之约 🎂 | 在一起365天 | 庆祝一年相伴 |
| 千年之恋 👑 | 在一起1000天 | 庆祝长久坚持 |

**设计理念：** 每个成就都是"事件驱动"的，不是"你做了X次就得成就"。这样避免了刷数据的行为，让每个成就都真正有意义。

### 3.9 设置页（pages/settings）

采用折叠面板设计，8个面板分工明确：

1. **双向绑定：** 生成邀请码 → 对方输入 → 绑定关系。包含前任数据检测。
2. **AI 称呼：** 设置小爱如何称呼两人（宝贝/老公/老婆...或自定义）
3. **纪念日：** 添加初吻、求婚、生日等自定义纪念日
4. **通知偏好：** 问候、周报、对方动态、纪念日提醒的开关
5. **背景音乐：** 3首内置BGM，支持切换/暂停
6. **隐私安全：** 隐私模式、截图警告、预览遮罩、分享遮罩
7. **回收站：** 查看已删除数据，30天保留期，支持恢复或彻底清空
8. **存储管理：** 查看缓存大小，清除图片缓存

### 3.10 情侣档案（pages/profile）

设置双方的基本信息：名字、头像、恋爱开始日期、城市等。支持 GPS 定位自动获取城市名（反向地理编码）。

## 四、AI Agent 核心机制 ★ 最重要

### 4.1 Agent 自主推理循环（ReAct 模式）

```
用户发消息
    ↓
小爱思考："用户想让我做什么？需要调工具吗？"
    ↓
调用工具 1（如：查约会记录）
    ↓
看到工具结果 → 再思考："还需要更多信息吗？"
    ↓
调用工具 2（如：查愿望清单）
    ↓
综合分析 → 生成最终回复
```

这个循环最多执行 5 轮，防止死循环。

### 4.2 10个 AI 可调用工具

| 工具名 | 功能 |
|--------|------|
| search_date_records | 搜索约会记录 |
| add_date_record | 添加约会记录 |
| get_wish_list | 查看愿望清单 |
| add_wish | 添加愿望 |
| get_moments | 查看心动瞬间 |
| add_moment | 记录心动瞬间 |
| get_promises | 查看约定列表 |
| add_promise | 添加约定 |
| generate_blessing | 生成祝福语 |
| plan_date | 策划约会方案 |

### 4.3 结构化记忆

小爱会从对话中自动提取关于情侣的信息：
- **偏好记忆：** "她喜欢喝奶茶" → 记下，下次推荐约会时提到奶茶店
- **里程碑记忆：** "我们第一次约会是在..." → 记下重要日期
- **矛盾记忆：** 吵架的原因 → 记下，调解时参考
- **愿望线索：** "想去海边" → 记下，加入愿望推荐

### 4.4 主动服务提醒（Nudge）

小爱不会被动等待，她会主动检查：
- 超过14天没约会 → 提醒策划约会
- 愿望积压 ≥ 3个 → 提醒去实现
- 纪念日临近（3天内）→ 提醒准备惊喜
- 约定超过7天没打卡 → 提醒履约

## 五、三大"雷区"设计 ★ 面试亮点

这是本项目最有深度的设计——不仅考虑功能，还考虑了真实情感场景中的风险。

### 雷区1：分手数据安全

**问题：** 如果情侣分手了，共同数据怎么办？如果新恋情开始，前任数据怎么处理？

**方案：**
- 解绑时提供两种选择：「保留个人副本」或「彻底清空」
- 保留模式：所有数据打上 `_era: 'archived'` 标签，变成个人回忆
- 删除模式：彻底清空所有共享记录
- 重新绑定时，检测是否有前任归档数据，提示用户确认清空
- 24小时反悔期：发起解绑后有24小时冷静期

### 雷区2：成就系统防刷设计

**问题：** 如何避免用户为了刷成就而制造虚假数据？

**方案：**
- 每个成就都是**里程碑事件**而非计数："记录第一次约会" vs "记录10次约会"
- 事件驱动触发：只有真正发生了这件事才解锁
- 不可逆转：解锁后不会因为删除数据而失去成就

### 雷区3：AI 调解双向同意

**问题：** AI 介入情侣矛盾前，如何保证双方都愿意？

**方案：**
- A 发起调解请求 → 生成会话ID
- B 输入会话ID确认加入 → 调解才能开始
- 双方都同意后，AI 以"情感翻译官"角色介入
- AI 调解铁律：不归因、不站队、不评判对错、绝不说"分手"
- 无 sessionId 的调解请求会被拒绝

## 六、安全设计

### 6.1 内容安全熔断

- **输入过滤：** 检测色情、暴力、政治敏感、自残等关键词，拦截并返回安全提示
- **输出过滤：** 即使 DeepSeek 返回了不当内容，也会被正则过滤替换
- **分级响应：** 根据严重程度返回不同的安全提示语

### 6.2 软删除 + 回收站

- 所有删除操作都是软删除（加 `_deletedAt` 标记）
- 30天保留期，期间可恢复
- 过期自动清理
- 支持一键清空回收站

### 6.3 隐私保护

- 隐私模式：首页名字隐藏，显示为"我们"
- 截图警告：检测到截图时弹出提醒
- 预览遮罩：多任务切换时模糊页面内容

## 七、前端小程序知识点总结

### 7.1 微信小程序框架核心概念

| 知识点 | 在项目中的应用 |
|--------|---------------|
| **Page() 注册页面** | 每个页面都是一个 Page 实例，包含 data、生命周期函数、事件处理函数 |
| **App() 全局应用** | app.js 管理全局数据（couple信息、settings），各页面通过 `getApp()` 获取 |
| **数据绑定** | WXML 中用 `{{变量名}}` 绑定数据，JS 中用 `this.setData()` 更新视图 |
| **条件渲染** | `wx:if="{{condition}}"` 控制元素显示/隐藏 |
| **列表渲染** | `wx:for="{{array}}"` 遍历数组，`wx:key` 指定唯一标识 |
| **事件绑定** | `bindtap`（点击）、`bindinput`（输入）、`bindconfirm`（确认）等 |
| **事件传参** | 通过 `data-xxx` 属性传参，在事件处理函数中用 `e.currentTarget.dataset.xxx` 获取 |

### 7.2 导航与路由

| 方式 | 使用场景 |
|------|---------|
| `wx.switchTab()` | 跳转到 tabBar 页面（首页、聊天、记忆馆、设置） |
| `wx.navigateTo()` | 跳转到非 tabBar 页面（愿望、瞬间、约定、成就等） |
| `wx.navigateBack()` | 返回上一页 |

### 7.3 网络通信

| 方式 | 使用场景 |

### 7.4 本地存储

## 九、项目亮点总结（面试这样说）

1. **AI Agent 不是简单的"问-答"，而是自主推理+多工具调用：** 小爱能自己决定调用什么工具、调用几次、按什么顺序，最多5轮推理循环

2. **流式输出体验：** WebSocket 实时推送，AI 回复像打字一样逐字出现，且做了8秒超时 HTTP 降级

3. **产品安全意识（三大雷区）：** 不是只写 CRUD，而是考虑了分手数据安全、成就防刷、AI 调解需双方同意等真实场景的边界情况

4. **全栈能力：** 前端小程序 + Node.js 后端 + 云函数 + AI API 对接，完整链路

5. **工程化思维：** 双模式运行（本地/云端一键切换）、软删除回收站、安全熔断、降级容错

6. **用户体验细节：** 骨架屏加载、触觉反馈、下拉刷新、隐私模式、背景音乐
