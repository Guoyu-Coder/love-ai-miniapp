## 一、演示路线图（面试时怎么展示）

### 推荐演示顺序（15-20分钟）

```
第一步：一句话介绍项目（30秒）
    ↓
第二步：打开首页，展示完整功能（2分钟）
    ↓
第三步：核心演示 —— AI 对话（5分钟）★ 最重要
    ↓
第四步：展示三大"雷区"设计（3分钟）
    ↓
第五步：展示设置页/回收站/成就（2分钟）
    ↓
第六步：打开代码，展示架构（3分钟）
    ↓
第七步：总结亮点，等老师提问（2分钟）
```

### 具体每一步怎么演示

#### 第一步：一句话介绍

> "我做的是一个微信小程序，叫「我们的空间」，它是一个**接入了 AI 大模型的情侣陪伴应用**。它不仅能聊天，还能自动调用工具帮情侣记录约会、管理愿望、生成报告，甚至在他们吵架时充当情感翻译官。"

**要点：** 一开口就把"AI大模型 + 实际产品"这两个关键词抛出来。

#### 第二步：首页演示

1. 打开首页，指着头部的名字和天数："这里展示情侣信息，在一起多少天，距离下一个纪念日还有多久"
2. 往下滑，指着灵感卡片："这里每天会有一条 AI 生成的约会灵感，点击可以刷新"
3. 指着数据统计："约会次数、愿望数、心动瞬间、约定数量，一目了然"
4. 指着成就墙："这是成就系统，但我的设计是**基于里程碑事件**的，不是刷次数"

#### 第三步：AI 对话演示 ★ 最重要

这是整个演示的核心，要展示三个层次：

**层次1 —— 普通对话：**
- 先发一条普通消息，比如"小爱你好呀"，看它回复
- 然后解释："它不只是聊天，它能调用工具"

**层次2 —— 工具调用：**
- 发一条"帮我记录一次约会，昨天在朝阳公园"
- **重点展示：** 展开"小爱的思考过程"面板
- 解释："你看，这里显示了 AI 的推理过程——它先思考我需要什么，然后决定调用'添加约会记录'这个工具，工具执行完成后告诉我结果。这就是 Agent 模式，AI 不是只会聊天，它**能做事**。"

**层次3 —— 多工具链：**
- 发一条"帮我做一下恋爱总结"
- 展示 AI 连续调用了多个工具（先查约会 → 再查愿望 → 综合回答）
- 解释："它能自主规划，先查什么再查什么，最多可以连续调用 5 轮工具。"

**额外展示 —— 流式输出：**
- 指出 AI 回复是**一个字一个字打出来的**，这是通过 WebSocket 实现的流式输出
- 如果 WebSocket 超时（8秒），会自动降级到 HTTP 请求

#### 第四步：三大雷区

1. **雷区1 —— 分手数据安全：**
   - 打开设置页 → 关系绑定 → 点击"断开绑定"
   - 展示两个选项：保留副本 vs 彻底清空
   - 解释："这是考虑了真实情感场景，分手后的数据处理"

2. **雷区2 —— 成就防刷：**
   - 打开成就页面
   - 解释："每个成就都是事件驱动的，比如'第一次约会'、'见家长'、'旅行'，不是'约会100次'这种可以刷的数量"

3. **雷区3 —— AI 调解需双方同意：**
   - 回到聊天页，点击"请求调解"
   - 解释："必须双方都同意，AI 才会介入，而且 AI 被设定了铁律——不归因、不站队、不说分手"

#### 第五步：展示其他功能

- 打开约会记忆馆，展示时间轴和照片墙两种视图
- 打开设置页，展示折叠面板、回收站、背景音乐

#### 第六步：代码展示

打开 VS Code，展示：

1. **项目结构：** 前端 miniprogram/ 目录，后端 server/ 目录，云函数 cloudfunctions/ 目录
2. **server.js：** 指出 SYSTEM_PROMPT（AI 人格定义）、agentLoop 函数（推理循环）、WebSocket 流式实现
3. **agent.js：** 前端的 Agent 客户端，多对话管理
4. **api.js：** 双模式切换（云函数 / 本地服务器），自动降级

## 三、常见面试问题及回答

### Q1："你这个 AI Agent 是怎么工作的？"

**回答思路：**

> "我用的是 ReAct 模式（Reasoning + Acting）。用户发来消息后，我把消息加上系统提示词、历史对话、以及结构化记忆一起发给 DeepSeek。DeepSeek 会返回两种结果：要么直接回复文字，要么返回一个 tool_call（函数调用）。
>
> 如果返回了 tool_call，比如它想调用 add_date_record，我就在后端执行这个函数（写数据库），然后把执行结果再发回给 DeepSeek，让它根据结果继续思考。这个循环最多跑 5 轮，防止死循环。
>
> 最终 DeepSeek 返回文字回复，我做安全过滤后展示给用户。整个过程用户能看到——我做了推理过程可视化。"

### Q2："Function Calling 是什么？你是怎么用的？"

**回答思路：**

> "Function Calling 是让大模型能调用外部工具的能力。我在代码里定义了 10 个工具，每个工具有名字、描述、参数 schema。比如 add_date_record 这个工具，我告诉 DeepSeek 它需要的参数是 date（日期）、activity（活动）、location（地点）这些。
>
> 当用户说'帮我记录昨天的约会'，DeepSeek 会返回一个 JSON：{ name: 'add_date_record', arguments: { date: '2025-05-24', activity: '看电影' } }。我在后端解析这个 JSON，执行对应的数据库操作，然后把结果告诉 DeepSeek。
>
> 关键点：DeepSeek 自己**决定**要不要调工具、调哪个、传什么参数——不是我写 if-else 判断的。"

### Q3："你这个项目的难点在哪？"

**回答思路（挑 3-4 个说）：**

> "第一，**Agent 推理循环的设计。** AI 调用工具后，工具返回的结果需要重新喂给 AI，AI 可能还要继续调工具。这个循环什么时候结束？最多几轮？每轮的结果怎么展示给用户？这些都需要仔细设计。
>
> 第二，**流式输出的实现。** 普通的 HTTP 请求要等 AI 全部生成完才返回，用户体验很差。我通过 WebSocket 实现了流式推送，AI 生成一个字就推一个字，像打字机一样。而且做了 8 秒超时自动降级到 HTTP 的容错机制。
>
> 第三，**产品安全的边界设计。** 不是只考虑功能能不能跑通，还考虑了真实场景中的风险——分手后数据怎么办、AI 调解需不需要双方同意、成就系统会不会被刷。这三个'雷区'方案是我自己思考并实现的。
>
> 第四，**双模式架构。** 开发环境用本地 Express 服务器调试方便，生产环境用微信云函数免运维。前端一行代码切换，而且做了降级——云函数挂了自动用本地服务器。"

### Q4："你遇到了什么问题？怎么解决的？"

**回答思路（挑几个具体的说）：**

> **问题1：云函数部署后超时**
> - 现象：云函数部署到微信云后，手机上调用一直超时
> - 排查：看了云函数日志，发现报 `fetch is not defined`
> - 原因：微信云函数的 Node.js 16 运行环境没有自带 `fetch` API（这是 Node 18+ 才有的）
> - 解决：在 package.json 里加 `node-fetch@^2.7.0` 依赖，用 `require('node-fetch')` 引入
>
> **问题2：API Key 硬编码的安全风险**
> - 现象：DeepSeek 的 API Key 直接写在代码里
> - 风险：如果代码提交到公开仓库，Key 会泄露，别人可以盗用
> - 解决：用 dotenv 把 Key 放到 .env 文件，.gitignore 忽略 .env，服务器启动时从环境变量读取。Key 泄露后立即在 DeepSeek 后台重新生成
>
> **问题3：Agent 消息丢失**
> - 现象：用户发的消息没有存入对话历史，导致 AI 缺少上下文
> - 原因：sendStream 从 conv.messages 读最后一条 user 消息，但 sendMessage 没有先把用户消息 push 进去
> - 解决：在 sendMessage 中，先 `conv.messages.push({ role: 'user', content: text })` 再调 sendStream
>
> **问题4：首页打开时空消息列表报错**
> - 现象：聊天页的 scroll-into-view 在消息为空时生成 `msg--1` 这个非法 ID
> - 解决：改成条件判断 `messages.length > 0 ? 'msg-' + (messages.length - 1) : ''`

### Q5："为什么选择 DeepSeek 而不是其他模型？"

**回答思路：**

> "几个原因：一是 DeepSeek 支持 Function Calling，这是实现 Agent 工具调用的基础；二是它有 OpenAI 兼容的 API 格式，开发和迁移都很方便；三是中文能力强，我这个项目面向的是中国用户；四是性价比高，作为学生项目预算友好。"

### Q6："你的数据库是怎么设计的？"

**回答思路：**

> "开发阶段我用的是内存数据库——一个 JS 对象，重启就没了。生产环境用的是微信云数据库（MongoDB 风格的文档数据库）。
>
> 数据模型主要有：couples（情侣档案）、date_records（约会）、wishes（愿望）、moments（瞬间）、promises（约定）、chat_history（聊天记录）、reports（报告）、achievements（成就）等十几个集合。
>
> 设计上有个特点：**软删除**。所有删除操作不是真删，而是加一个 `_deletedAt` 时间戳，数据进回收站保留 30 天。这样用户误删了还能恢复。"

### Q7："你这个项目有什么不足？如果继续做你会怎么做？"

**回答思路（诚实地展示思考深度）：**

> "目前有几个可以改进的地方：
> 1. **没有做用户认证**——现在是单用户模式，靠微信 OPENID 区分用户，但没有做登录态管理
> 2. **没有做消息推送**——提醒功能只能在小程序内看到，不能通过微信服务通知推送给用户
> 3. **WebSocket 没有做重连机制**——如果连接断了，需要手动刷新页面重连
> 4. **云函数和本地服务器的代码有重复**——两边都有 agentLoop 的实现，维护成本高。理想的是抽一个公共模块
> 5. **没有做测试**——目前靠手动测试，应该加上单元测试和接口测试
>
> 如果继续做，我优先做：消息推送（用户体验提升最大）和 WebSocket 自动重连。"

### Q8："说一下你对 AI Agent 的理解？"

**回答思路：**

> "我认为 AI Agent 和普通聊天机器人的核心区别是：**Agent 能做事，不只是会说话。**
>
> 普通聊天机器人：你问它答，一问一答。
> AI Agent：你给它一个目标，它自己决定怎么做——调用什么工具、按什么顺序、中间怎么调整。
>
> 在我的项目里，小爱就是一个 Agent。用户说'帮我策划一个周末约会并记录下来'，小爱会：
> 1. 先调 plan_date 生成约会方案
> 2. 把方案展示给用户
> 3. 如果用户说'可以，记下来'，再调 add_date_record 写入数据库
>
> 整个过程中，不是我写代码告诉它'先做A再做B'，而是它自己推理出这个步骤。这就是 Agent 的自主性。"

## 四、如何在演示中突出"AI 大模型能力"

### 关键话术

**开篇时：**
> "这个项目最大的特点是，它不只是调一个 AI 接口来聊天，而是让 AI 成为了一个**具有自主行动能力的 Agent**。"

**展示工具调用时：**
> "你看这里，我什么都没做，只是说了一句话，AI 自己判断要调哪个工具、传什么参数。这不是我写了 if-else 判断关键词，而是 AI 的推理能力。"

**展示推理过程时：**
> "这里能看到 AI 的思考过程——它先想'用户想让我记录约会，我需要调 add_date_record'，然后执行，看到结果后继续生成回复。这个叫 **Chain of Thought**，是可解释 AI 的一部分。"

**展示多工具链时：**
> "刚才它连续调了两个工具，先查约会再查愿望，最后综合分析。这是**多步推理**——AI 自己规划了执行顺序，不是我做的工作流编排。"

**总结时：**
> "总结一下，这个项目展示了三个层次的 AI 应用能力：
> 1. **基础层：** 接入大模型 API，进行自然语言对话
> 2. **Agent 层：** 让 AI 自主调用工具、操作数据库、完成实际任务
> 3. **产品层：** 把 AI 能力包装成真正可用的产品，考虑了安全、体验、边界情况"

## 五、常见追问及应对

### 追问1："你怎么保证 AI 不会乱说话？"（安全相关）

> "我做了三层防护：
> 1. **系统提示词约束：** 在发给 AI 的 system prompt 里明确写了安全红线——不输出色情/暴力/政治内容，不建议分手
> 2. **代码层正则过滤：** 用户输入和 AI 输出都经过正则匹配检查，命中敏感词就拦截
> 3. **分级响应：** 不同严重程度的内容用不同的安全提示语回应用户
>
> 这是**纵深防御**的思路——不能只靠 AI 自觉，代码层也要兜底。"

### 追问2："Function Calling 失败了怎么办？"

> "我在代码里做了错误处理：
> 1. 如果 AI 调用了一个不存在的工具，会返回'未知工具'的错误，让 AI 重新思考
> 2. 工具执行失败了（比如读数据库出错），会把错误信息返回给 AI，让它尝试其他方式
> 3. 整个循环有 5 轮上限，防止 AI 陷入死循环
> 4. 最坏情况下，AI 会回复一条友好的兜底消息"

### 追问3："你怎么处理高并发？"

> "诚实地讲，目前这个项目是面向个人的，没有做高并发优化。但我了解一些思路：
> 1. 云函数本身就是 Serverless，微信云会自动扩容
> 2. DeepSeek API 调用是瓶颈，可以加请求队列或缓存常见回复
> 3. 如果用户量大，可以把聊天历史从内存移到 Redis
>
> 这些是我下一步要学习的方向。"

### 追问4："你觉得这个项目的商业价值在哪？"

> "情侣市场是一个被验证的付费场景。类似的 App 比如'恋爱记'、'小恩爱'都有不错的用户量。
>
> 我这个项目的差异化在于 AI——它不是简单的工具集合，而是一个有人格、能主动服务的 AI 伴侣。这能提升用户粘性。商业模式上可以考虑订阅制（AI 功能付费）或者增值服务（情侣周边、纪念日策划等）。
>
> 当然，目前还是个学习项目，重点是展示我的技术能力。"

## 六、我遇到的问题和解决过程（完整版）

这是面试时最容易得分的问题——"你在项目中遇到过什么问题？"

### 问题1：云函数部署后超时（技术深度）

**现象：** 把 agentCore 云函数部署到微信云后，手机上测试一直超时，控制台报错但看不到具体原因。

**排查过程：**
1. 先在本地服务器测试，功能正常 → 排除代码逻辑问题
2. 检查云函数日志，发现 `fetch is not defined`
3. 查资料发现微信云函数用的是 Node.js 16，而 `fetch` 是 Node 18 才内置的 API

**解决方案：**
- 在 `package.json` 添加依赖 `node-fetch: ^2.7.0`
- 在云函数顶部 `const fetch = require('node-fetch')`
- 注意必须用 v2（CJS），v3 是 ESM 在云函数环境不兼容

**学到的：** 云环境和本地环境的差异要注意，Node 版本、内置 API、网络限制都可能不一样。

### 问题2：API Key 安全（安全意识）

**现象：** 最开始我把 DeepSeek API Key 硬编码在代码里 `const DEEPSEEK_KEY = 'sk-xxx'`。

**问题：** 如果代码提交到 GitHub 公开仓库，Key 会被爬虫扫描到，被他人盗用产生费用。

**解决过程：**
1. 安装 dotenv → `require('dotenv').config()`
2. 创建 `.env` 文件，写入 `DEEPSEEK_KEY=sk-xxx`
3. `.gitignore` 添加 `.env`
4. 代码改为 `const DEEPSEEK_KEY = process.env.DEEPSEEK_KEY`，启动时检查是否为空
5. 泄露的 Key 立刻在 DeepSeek 后台重新生成（revoke + regenerate）

**学到的：** 敏感信息必须用环境变量管理，这是后端开发的基本素养。

### 问题3：语法错误导致模块加载失败（调试能力）

**现象：** 微信开发者工具报 agent.js 模块加载失败，`module not defined`。

**排查过程：**
1. 用 `node --check agent.js` 检查语法 → 发现两处 class 方法后面多了逗号 `  },`
2. JavaScript class 语法中方法之间不能有逗号（和对象字面量不同）

**解决方案：** 删除多余逗号，`node --check` 确认通过。

**学到的：** 写代码时要注意 class 和 object 的语法差异，可以用 `node --check` 做语法校验。

### 问题4：用户消息没有存入对话历史（逻辑 Bug）

**现象：** AI 聊天的上下文不完整，AI 好像不知道用户刚说了什么。

**排查过程：**
1. 检查对话历史数据 → 发现用户消息缺失
2. trace 代码：`sendStream()` 从 `conv.messages` 读最后一条 user 消息，但 `sendMessage()` 没先 push 用户消息

**解决方案：** 在 `sendMessage()` 调用 `sendStream()` 之前，先 `conv.messages.push({ role: 'user', content: text })`

**学到的：** 数据流要完整追踪——前端显示的消息和传给 AI 的历史要分别管理，不能只顾一头。

### 问题5：首页空状态时的 WXML 模板错误

**现象：** 聊天页打开时，`scroll-into-view` 属性值为 `msg--1`（因为 messages 数组为空，`length - 1 = -1`），这是一个无效的 ID 引用。

**解决方案：** 改为条件表达式 `{{messages.length > 0 ? 'msg-' + (messages.length - 1) : ''}}`，空数组时传空字符串不触发滚动。

### 问题6：服务器端口冲突

**现象：** 启动 server.js 报 `EADDRINUSE` 错误，端口 3001 被占用。

**解决：** `taskkill //f //im node.exe` 杀掉残留进程后重新启动。

**学到的：** 开发中进程管理很重要，特别是 WebSocket 连接不会随页面刷新自动断开。
